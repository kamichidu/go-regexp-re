= Bit-parallel DFA (BP-DFA)
:sectnums:

== Concept

The BP-DFA is a specialized, branchless execution path implemented in Pure Go for patterns with 64 or fewer states.
By leveraging bitwise operations on `uint64` vectors, it eliminates memory loads and executes almost entirely within CPU registers, targeting the theoretical speed limit of string matching on modern hardware.

== Bit-parallel State Representation

The BP-DFA uses a single `uint64` to represent the active set of NFA states. Each bit corresponds to a specific NFA state, with the bit position defining its priority.

[source]
----
 uint64 State Vector:
 MSB (Bit 63)                                          LSB (Bit 0)
  |                                                     |
 [ ] [ ] [ ] ... [ ] [ ] [ ] [ ] [ ] [ ] [ ] [ ] [ ] [ ] [1]
  |               |   |   |   |   |   |   |   |   |   |   |
  +-------+-------+---+---+---+---+---+---+---+---+---+---+
          |           |   |   |   |   |   |   |   |   |   |
      Low Priority    |   |   |   |   |   |   |   |   | Higher Priority
                      |   |   |   |   |   |   |   |   | (Leftmost-First)
                      |   |   |   |   |   |   |   |   +-- Start State / Prio 0
                      |   |   |   |   |   |   |   +------ State 1 / Prio 1
                      |   |   |   |   |   |   +---------- State 2 / Prio 2
----

== Core Transition Logic

The transition is performed by splitting the 64-bit state vector into 8-bit chunks and using pre-calculated successor tables. This allows for constant-time transitions even for complex NFA graphs with loops and alternations.

[source]
----
1. Initialize State: state = bp.StartMasks[initialContext]
   The starting state is precalculated for each of the 64 possible empty-width contexts.

2. Character Masking: active = state & bp.CharMasks[input[i]]
   Filters the active paths based on the current input byte.

3. Successor Transition: next = bp.SuccessorTable[0][active & 0xFF] | ... | bp.SuccessorTable[7][(active >> 56) & 0xFF]
   Computes the next set of reachable states by following all epsilon transitions simultaneously.

4. Context Filtering: next &= bp.ContextMasks[nextContext]
   Filters paths based on anchor conditions (e.g., ^, $, \b) at the resulting junction.

5. Match Detection: (active & bp.MatchMasks[nextContext]) != 0
   Detects if any path that matched the current byte satisfies the match condition in the junction context.
----

== Key Technologies

=== 1. Context-Aware Vector Pruning
The BP-DFA uses pre-compiled `ContextMasks [64]uint64` and `MatchMasks [64]uint64` to resolve all 64 possible junction contexts in $O(1)$.
* **ContextMasks**: Prunes the state vector at each junction, killing paths that violate anchors.
* **MatchMasks**: Identifies which states are considered "accepting" at the current junction.

=== 2. Chunked Successor Lookups
To handle general NFA graphs that cannot be expressed as simple shifts, the engine uses a chunked lookup table (`SuccessorTable [8][256]uint64`). This technique enables branchless transitions across the entire 64-bit state vector with only 8 memory loads (likely hitting L1) and 7 OR operations.

== Strategy Selection & SIMD Warp

Small patterns (up to 64 instructions) are automatically specialized for BP-DFA.

* **SIMD Warp Priority**: Even if a pattern is small enough for BP-DFA, it may be delegated to the table-based DFA if it has a long constant prefix (>= 4 bytes). This allows the engine to use SIMD acceleration (`bytes.Index`) to skip non-matching segments, which often outperforms bit-parallel scanning.
* **Full Anchor Support**: Unlike traditional bit-parallel implementations, this engine supports anchors like `\b` and multiline `^`/`$` within the BP-DFA loop, providing consistent high performance for small complex patterns.

== Advantages

* **Pure Go High Performance**: Achieves performance comparable to low-level C implementations by aligning perfectly with the Go compiler's register allocation.
* **L1 Cache Optimization**: Successor tables and masks fit within the L1 data cache (approx. 16KB).
* **Deterministic Execution**: Guarantees a constant instruction count per byte, providing predictable and low-latency matching.
