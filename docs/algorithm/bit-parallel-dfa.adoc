= Bit-parallel DFA (BP-DFA)
:sectnums:

== Concept

The BP-DFA is a specialized, branchless execution path implemented in Pure Go for patterns with 64 or fewer states.
By leveraging bitwise operations on `uint64` vectors, it eliminates memory loads and executes almost entirely within CPU registers, targeting the theoretical speed limit of string matching on modern hardware.

== Bit-parallel State Representation

The BP-DFA uses a single `uint64` to represent the active set of NFA states. Each bit corresponds to a specific NFA state, with the bit position defining its priority.

[source]
----
 uint64 State Vector:
 MSB (Bit 63)                                          LSB (Bit 0)
  |                                                     |
 [ ] [ ] [ ] ... [ ] [ ] [ ] [ ] [ ] [ ] [ ] [ ] [ ] [ ] [1]
  |               |   |   |   |   |   |   |   |   |   |   |
  +-------+-------+---+---+---+---+---+---+---+---+---+---+
          |           |   |   |   |   |   |   |   |   |   |
      Low Priority    |   |   |   |   |   |   |   |   | Higher Priority
                      |   |   |   |   |   |   |   |   | (Leftmost-First)
                      |   |   |   |   |   |   |   |   +-- Start State / Prio 0
                      |   |   |   |   |   |   |   +------ State 1 / Prio 1
                      |   |   |   |   |   |   +---------- State 2 / Prio 2
----

== Core Transition Logic

The transition is performed by shifting the state vector and applying a character mask. This allows the engine to track all possible matching paths simultaneously in constant time.

[source]
----
1. Initialize State: state = bp.StartMasks[initialContext]
   The starting state is precalculated for each of the 64 possible empty-width contexts (e.g., ^, \b).

2. Shift & Insert: (state << 1 | 1)
   Moves all active paths forward and seeds a new match at the start state.
   (Note: For unanchored search, the start mask is re-applied or state is seeded).

3. Masking: & masks[input[i]]
   Filters the active paths based on the current input byte.

4. Context Filtering: & bp.ContextMasks[currContext]
   Filters paths based on anchor conditions (e.g., ^, $, \b).

5. Match Detection: (state & bp.MatchMasks[currContext]) != 0
   Uses precalculated context-dependent match masks to detect if any path reached a match state.
----

== Key Technologies

=== 1. Zero-Traverse Anchor Resolution
Rather than using runtime NFA traversals or virtual bytes, the BP-DFA uses pre-compiled `StartMasks [64]uint64` and `ContextMasks [64]uint64`. 
* **StartMasks**: Capture the epsilon closure of the starting position, considering all 64 possible boundary conditions at the first byte.
* **ContextMasks**: Aggregates the character classes and boundary conditions into a single bitwise AND, resolving anchors like `\b`, `^`, and `$` with $O(1)$ cost.

=== 2. Winning Absolute Priority
In a bit-parallel environment, multiple NFA paths can overlap. The engine resolves submatch priorities at the bit-level, ensuring that the "absolute winning path" is determined deterministically without the overhead of tracking individual path histories at runtime.

== Strategy Selection & SIMD Warp

While BP-DFA is extremely fast for small patterns, it does not support SIMD-accelerated skipping (SIMD Warp) because the bitset must be updated for every byte. 

* **Anchor-Heavy Patterns**: Patterns with anchors like `\bGET\b` are now **delegated to the table-based DFA** (Strategy 4) even if they are small enough for BP-DFA. This allows the engine to use `bytes.Index` to skip non-matching segments at GB/s speeds, which is significantly faster than bit-parallel scanning for sparse matches.
* **Shortcut Heuristic**: The "Architectural Shortcut" only selects BP-DFA for patterns that are simple AND anchor-free.

== Advantages

* **Pure Go High Performance**: Achieves performance comparable to low-level C implementations by aligning perfectly with the Go compiler's register allocation.
* **L1 Cache Optimization**: Successor tables and masks fit within the L1 data cache (approx. 16KB).
* **Deterministic Execution**: Guarantees a constant instruction count per byte, providing predictable and low-latency matching.
