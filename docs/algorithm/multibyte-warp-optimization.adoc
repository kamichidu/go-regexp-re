= Multi-byte Warp Optimization & State Explosion Protection

This document describes the design and implementation of two synergistic optimizations: the **Multi-byte Warp** for $O(1)$ character throughput, and **Priority-based Uniquing** for state explosion protection.

== 1. Objective

* **Prevent State Explosion**: Avoid the combinatorial explosion of DFA states caused by loop structures (`.*`, `a*`) where redundant NFA paths with different priorities diverge.
* **Maximize Throughput**: Achieve near-constant time per UTF-8 character by "warping" (skipping) trailing bytes after processing only the Lead Byte.
* **Maintain Precision**: Ensure that capture groups and anchors remain accurate even when byte-level scanning is partially bypassed.

== 2. Multi-byte Warp (External Delegation Skip)

The Warp strategy treats a multi-byte UTF-8 sequence as a single logical transition in the DFA hot loop.

=== 2.1 Lead-Byte Detection & Length Calculation

The total length $L$ of a UTF-8 character can be determined from its first byte (Lead Byte). For invalid UTF-8 bytes (e.g., continuation bytes appearing as lead bytes), the length is treated as 1 to match Go's standard library behavior.

[cols="1,1,1"]
|===
| Bit Pattern | $L$ (Total Bytes) | $N$ (Trailing Bytes to Skip)
| `0xxxxxxx`  | 1                 | 0 (ASCII)
| `110xxxxx`  | 2                 | 1 (Valid Lead)
| `1110xxxx`  | 3                 | 2 (Valid Lead)
| `11110xxx`  | 4                 | 3 (Valid Lead)
| Others      | 1                 | 0 (Invalid Lead/Continuation)
|===

The execution engine uses a robust `ir.GetTrailingByteCount(byteVal)` which employs range checks to correctly identify the skip distance $N$ and avoid over-skipping on invalid UTF-8 sequences.

=== 2.2 Execution Engine Logic (`matchExecLoop` / `submatchExecLoop`)

When the engine encounters a Lead Byte with `WarpStateFlag` (Bit 23) set in the transition table:

[source,go]
----
// Logic inside DFA execution loop
if (rawNext & ir.WarpStateFlag) != 0 {
    skip := ir.GetTrailingByteCount(byteVal)

    // For submatch extraction, we fill the history for intermediate bytes
    for j := 1; j <= skip; j++ {
        mc.history[i+j] = state & ir.StateIDMask
    }
    i += 1 + skip
}
----

The `WarpStateFlag` is applied **ONLY** when all active NFA paths in a state accept any valid UTF-8 continuation (e.g., `InstRuneAny` or `.` dot). Fixed multi-byte runes (like "あ") do not trigger Warp to ensure each byte is correctly verified.

=== 2.3 Byte-level Trie Expansion

During DFA construction, rune-based instructions are expanded into a series of byte transitions using a **UTF-8 Trie**. This ensures that:

1. **Precision**: All rune boundaries and character class members are correctly separated at the byte level.
2. **Determinism**: The DFA correctly handles overlapping byte prefixes (e.g., "あ" and "い" share the same lead byte `0xE3`).
3. **Standard Compatibility**: `.` (dot) matches any valid UTF-8 character by following the trie paths.

== 3. State Explosion Protection (Priority Uniquing)

State explosion occurs when loop structures create a vast set of NFA paths with diverging priorities.

=== 3.1 Winner-Takes-All Policy

In `epsilonClosureWithPathTags`, the engine enforces a strict uniquing rule based on the **Leftmost-First** principle:

* For each NFA state `(StateID, NodeID)`, only the **highest priority (minimum Priority value)** path is retained.
* Any lower-priority path reaching the same NFA state is discarded immediately, as it can never "win" the match.
* This convergence prevents the creation of thousands of DFA states for simple patterns like `.*` or `a*`.

=== 3.2 isBestMatch Optimization

A match is only finalized (`isBestMatch = true`) if **no other active NFA path** with equal or better priority can still reach a match. This ensures the engine correctly identifies the longest match in greedy repetitions.

== 4. Implementation Details

* **WarpStateFlag (Bit 23)**: Marks transitions that cover a full UTF-8 character.
* **StateIDMask (0x007FFFFF)**: Reserves bits for flags while allowing 8M states.
* **Minimization Preservation**: The `minimize` function uses bitwise operations to ensure `WarpStateFlag` and `TaggedStateFlag` are carried over to the minimized transition table.
* **Memory Tracking**: Construction caches (`nfaToDfa`, `closureCache`) are tracked against `MaxMemory` to prevent OOM.

=== Ambiguous Dot Junctions

Previously, placing a dot `.` between multi-byte characters was a known limitation. With the introduction of **Byte-level Trie Expansion**, this is no longer an issue. The DFA now correctly expands the dot into its constituent byte-level transitions, allowing for seamless transitions through any combination of single-byte and multi-byte characters with 100% standard library precision.
