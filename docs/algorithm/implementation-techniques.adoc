= High-Efficiency Implementation Techniques
:sectnums:

== Concept

Beyond architectural algorithms, `go-regexp-re` employs advanced implementation-level techniques specifically tuned for the Go compiler and runtime to achieve maximum hardware utilization.

== Branchless State ID Packing

The transition table stores "Packed State IDs" as 32-bit unsigned integers. These are not simple pointers; they are bit-fields containing both the target state and operational flags.

[source]
----
 32-bit Packed State ID (Canonical Layout):
 [31] [30] [29] [28] [27-22] [21] [20] [19-0]
  |    |    |    |      |      |    |    |
  |    |    |    |      |      |    |    +--- State Index (up to 1,048,576 states)
  |    |    |    |      |      |    |
  |    |    |    |      |      |    +-------- Reserved
  |    |    |    |      |      |
  |    |    |    |      |      +------------- WarpStateFlag (UTF-8 skip)
  |    |    |    |      |
  |    |    |    |      +-------------------- Anchor Mask (6-bit EmptyOp)
  |    |    |    |
  |    |    |    +--------------------------- CCWarpFlag (SWAR Warp possible)
  |    |    |
  |    |    +-------------------------------- AcceptingStateFlag (Match found)
  |    |
  |    +------------------------------------- AnchorVerifyFlag (Context check needed)
  |
  +------------------------------------------ TaggedStateFlag (Priority/Tag updates)
----

== Key Techniques

=== 1. In-State Acceptance Detection
Traditionally, a DFA engine lookups a separate boolean array to determine if a state is "accepting."
* **Technique**: `go-regexp-re` embeds the `AcceptingStateFlag` (Bit 29) directly into the state ID.
* **Benefit**: This allows the execution loop to detect matches using a bitwise AND on the existing state register, eliminating a memory load and a potential cache miss per byte.

=== 2. Reusable Register Context (matchContext Pool)
Submatch extraction requires a results slice (`[]int`). Allocating this on every match is a major performance bottleneck.
* **Technique**: The engine uses a `sync.Pool` of `matchContext` objects. Each context contains a pre-allocated `regsBuf [32]int` (supporting up to 15 subexpressions) and can dynamically grow for larger patterns.
* **Benefit**: This eliminates per-match heap allocations for most patterns, reducing GC pressure and stabilizing performance in high-throughput environments.

=== 3. Early Exit Optimization (Strict Greedy Finality)
To minimize redundant scanning, the engine employs an early-exit strategy called `IsBestMatch`. However, to remain compatible with Go's greedy matching rules (e.g., `a*` matching `aaa` rather than an empty string), this exit must be strictly guarded.

*   **Logic**: A state is marked as having a "Best Match" if and only if:
    `MatchPriority(state) <= MinPriority(state)`
*   **Significance**: `MinPriority` represents the absolute highest priority (lowest numerical value) that can be reached from the current state through any future transitions. If the current match already meets this threshold, no further scanning can improve the result, allowing for a safe $O(1)$ exit.
*   **Greedy Preservation**: In a loop like `a*`, the exit transition typically has a lower priority (higher value) than the loop transition. Therefore, `IsBestMatch` remains false until the loop can no longer be taken, ensuring the longest possible match.

=== 4. Hot-Loop Monomorphization (Manual Inlining)
The engine provides three physically distinct execution loops: `bitParallelExecLoop` (deprecated), `fastExecLoop`, and `extendedExecLoop`.
* **Problem**: Go generics (introduced in 1.18) can introduce runtime dictionary lookups for function shapes (`GCShape`), and standard polymorphism can prevent aggressive constant folding.
* **Solution**: By manually duplicating and specializing these loops (Monomorphization), the engine ensures that the Go compiler can:
  - Completely eliminate unreachable branches (e.g., `if hasAnchors`) during compilation.
  - Apply aggressive register allocation specific to the complexity of the loop.
  - Avoid any runtime indirection within the most critical performance path.
* **Warp Specialization**: This principle is also applied to **SWAR Warp Kernels**. Instead of checking the kernel type (`switch`) inside the 8-byte skip loop, the `switch` is performed once before entering a specialized, kernel-specific loop.

=== 5. Byte-Level UTF-8 Trie Expansion
Unlike many engines that decode UTF-8 into runes (`int32`) during execution, `go-regexp-re` processes raw bytes.
* **Mechanism**: During the NFA-to-DFA conversion, any non-ASCII character (UTF-8) is expanded into a byte-level Trie. This Trie is then "baked" directly into the DFA transition table.
* **Benefit**: This eliminates the overhead of `utf8.DecodeRune` (which involves complex branching and bit shifting) and allows multi-byte characters to be matched with the same $O(1)$ per byte performance as ASCII characters.

=== 6. Absolute Coordinate Context (Stability & Semantic Correctness)
To handle anchor verification and multi-pass submatch extraction correctly regardless of where a scan starts, the engine utilizes an **Absolute Coordinate Context** via the `ir.Input` structure.

*   **Semantic Correctness**: Anchors like `^` and `$` must identify the "True Start/End of Text." By propagating `AbsPos` (the offset of the current slice from the original input), verification functions can determine truth using simple arithmetic: `(AbsPos + i) == 0`. This eliminates the need for complex state flags to track "Am I at the real beginning?"
*   **Coordinate Stability**: In the 3-pass TDFA, results must be absolute. By using a single coordinate system (Absolute) throughout all passes, the engine avoids the overhead and bug-proneness of converting between "slice-relative" and "original-relative" indices during Pass 2 (Backtrace) and Pass 3 (Tagging).
*   **Zero-Copy Resumption**: When a pre-filter (MAP) identifies a match candidate in the middle of a buffer, the engine can resume DFA scanning using the same `ir.Input` without needing to physically slice the byte array, preserving context for junction verification (`\b`, etc.).

=== 7. Bitmask-First Short-Circuiting
Anchor verification functions (`VerifyBegin`, `VerifyEnd`, `VerifyWord`) are critical points in the hot loop. Even when they return `true`, the cost of their internal branching can be significant.

* **Technique**: Every verification function MUST employ a single-instruction bitmask check at the very beginning to detect if any relevant flags are present in the request (`req`).
[source,go]
----
func VerifyBegin(in Input, i int, req syntax.EmptyOp) bool {
    // Single-instruction bitmask check
    if (req & (syntax.EmptyBeginText | syntax.EmptyBeginLine)) == 0 {
        return true
    }
    // Detailed logic using in.AbsPos + i...
}
----
* **Impact**: Field evidence showed that omitting this unified check in favor of individual flag checks (even if mathematically equivalent) causes a **30-60% throughput regression**. This is due to increased instruction count and branch misprediction in the CPU pipeline when processing transitions that only require subset of anchors (e.g., a transition requiring only `\b` still calls `VerifyBegin`).

=== 8. Intelligent SWAR Pre-filtering
While `bytes.Index` provides SIMD-accelerated prefix scanning, many regex patterns (e.g., `[0-9]+`) do not have a constant literal prefix.
* **Technique**: The engine extracts the "Start Set" (all possible first bytes) of the pattern. If this set is sufficiently small (< 64 bytes), a specialized **SearchWarp** is generated.
* **Mechanism**: At the start of a match attempt, if the current DFA state is the `searchState`, the engine uses SWAR kernels to skip blocks of 8 bytes that do not contain any byte from the Start Set.
* **Safety**: This technique is automatically disabled for broad patterns (like `.*`) where the Start Set is too large, avoiding unnecessary overhead.

== Summary of Performance Gains

These techniques ensure that the "theoretical" $O(n)$ performance of the DFA becomes a "physical" reality, minimizing the "constant factor" cost to just a handful of CPU cycles per byte.
