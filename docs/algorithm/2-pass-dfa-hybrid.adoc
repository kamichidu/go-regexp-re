= DFA-First Hybrid (2-Pass Strategy)
:sectnums:

== Concept

DFAs are traditionally poor at extracting submatches (capturing groups). This is because when multiple NFA paths merge into a single DFA state, the history of which path "won" is lost. To extract submatches identical to the Go standard library, this project employs a "2-Pass Strategy."

== The Two Phases

=== Phase 1: Boundary Discovery
* **Objective**: Find the `[start, end]` boundaries of a match as quickly as possible.
* **Mechanism**: Uses a "pure DFA loop" or a loop with minimal tagging.

=== Phase 2: Localized Forward Scan
* **Objective**: Extract precise submatch information within the confirmed boundary.
* **Mechanism**:
  - **Principal DFA Rescan**: Performs a forward scan using the primary DFA. During the scan, capture registers (`regs`) are updated according to the priority-aware `applyTags` logic.
  - **BP-DFA Handover (Greedy Arbitration)**: When the scan reaches the `end` position, and the pattern involves greedy operators, the engine hands over the DFA state to the BP-DFA.
  - **Winning Bit Selection**: The BP-DFA converts the final DFA state into a bitmask to identify the "absolute winning NFA path" (the bit that reached `InstMatch` at exactly the `end` position).
  - **Greedy Tag Refinement**: Using pre-calculated `greedyTags`, the engine ensures that only tags belonging to greedy operators are overridden by the BP-DFA's terminal arbitration. This resolves the 1-byte ambiguity inherent in DFA greedy loops without affecting non-greedy captures.
  - **Zero-NFA Mandate**: This hybrid approach completely eliminates the need for a backtracking or thread-managed NFA rescan, maintaining pure $O(n)$ time complexity and zero runtime heap allocations.

== Key Technology: Winning Bit Arbitration
The core challenge in DFA submatch extraction is "Greedy Overrun." In a pattern like `(a*)b`, the DFA state at the match end might simultaneously contain a path that matched `a*` and a path that moved to `b`.

By bridging the DFA state with a BP-DFA bitmask (`uint64`), the engine can:
1. **Materialize the Set**: Convert the abstract DFA state ID back into its constituent NFA states (bits).
2. **Identify the Victor**: Select the highest priority bit (leftmost-first) that is in a matching state at the exact `end` coordinate.
3. **Correct the Boundary**: Precisely finalize capture indices based on this specific winning path, ensuring 100% compatibility with Go's `regexp` package.

== Advantages

* **$O(n)$ Guarantees**: All steps stay within a constant multiple of the input length; no backtracking.
* **Memory Bandwidth Savings**: Minimizes writes in Phase 1, keeping CPU caches hot for the data in Phase 2.
* **Full Standard Compatibility**: Guarantees results identical to the `regexp` package, even for complex patterns like `a*(a)`.
