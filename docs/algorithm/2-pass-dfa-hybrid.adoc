= DFA-Guided Bit-Parallel Traceback (DASH-BT)
:sectnums:

== Concept

DFAs are traditionally poor at extracting submatches (capturing groups) because the history of which specific NFA path was taken is lost during state merging. To achieve 100% Go standard library compatibility while maintaining $O(n)$ performance and avoiding NFA thread management (PikeVM), this project employs a **DFA-Guided Bit-Parallel Traceback (DASH-BT)** strategy.

This strategy treats the DFA state history as a "safe corridor" through the NFA's potential state space, allowing for a deterministic reconstruction of the unique winning path using bitwise operations.

== The Three Phases

=== Phase 1: Consolidated Discovery & Recording (Forward Pass)
* **Objective**: Find the `[start, end]` match boundaries and record the execution history in a single pass.
* **Mechanism**: Uses a specialized `submatchExecLoop` that performs standard DFA scanning while recording the `ir.StateID` at every byte position into a pre-allocated history buffer.
* **Lightweight History**: Records only 4-byte `ir.StateID`s instead of heavy NFA state sets.
* **Performance**: For common match sizes (e.g., <1024 bytes), the history fits entirely on the CPU stack (`[1024]int32`), eliminating heap allocations.

=== Phase 2: DFA-Guided Path Reconstruction (Backward Pass)
* **Objective**: Identify the unique sequence of NFA instructions taken for every byte of the match.
* **Mechanism**:
    - **Winning Node Identification**: At the `end` position, the engine selects the NFA node that reached a `Match` instruction with the highest absolute priority (leftmost-first) from the final DFA state's closure.
    - **Guided Reverse Trace**: Moving from `end` back to `start`, the engine consults the historical `ir.StateID`s. For each byte, it fetches the corresponding NFA closure from storage.
    - **Transition Resolution**: It identifies the unique predecessor `ir.NFAPath` (Instruction ID + Trie Node ID) that could have transitioned to the current node given the byte at that position.
    - **UTF-8 Precision**: By tracking both the Instruction ID and the Trie Node ID, the engine correctly handles multi-byte UTF-8 runes, even when a single NFA instruction spans multiple bytes.

=== Phase 3: Forward Recap (Tag Application)
* **Objective**: Apply capturing group tags along the confirmed path.
* **Mechanism**:
    - The engine walks forward along the `ir.NFAState` path reconstructed in Phase 2.
    - **Epsilon-Path Recovery**: Between byte-consuming transitions, the engine follows the highest-priority epsilon path (following `InstCapture`, `InstAlt`, etc.) to apply tags.
    - **Bit-Perfect Results**: This ensures the engine captures exactly the same boundaries as Go's `regexp` package, including edge cases like nested empty matches.

== Key Technology: On-Demand Closure Retrieval

=== Lightweight History
The engine only records 4-byte `ir.StateID`s during the forward scan. This keeps the history buffer small enough to fit in the CPU cache or on the stack for common match lengths.

=== Absolute Priority Maintenance
Unlike traditional DFA minimization which might lose path priority, this engine preserves absolute NFA priorities during state construction. This allows Phase 2 to always pick the "winning" NFA path that Go's leftmost-first semantics demand.

=== Zero-Allocation Discovery
During the forward scan, no heap allocations occur. The NFA closures required for traceback are retrieved from a pre-calculated storage (memory or file-backed), ensuring that submatch extraction is both fast and memory-stable.

=== Tiled Checkpoint Scanning (Future)
For extremely long inputs where the history exceeds stack limits, the engine uses 512-byte "Checkpoints". It records only periodic `StateID`s and re-scans local blocks on-demand, keeping the working set within the L1 cache.

== Advantages

* **$O(n)$ Complexity**: Both recording and traceback are strictly linear with respect to match length.
* **Stack-Resident Memory**: Minimizes memory bandwidth by reducing history size by 6x.
* **Bit-Perfect Compatibility**: Successfully handles complex cases like greedy loops, non-greedy overlaps, and nested empty matches by mirroring Go's priority rules.
