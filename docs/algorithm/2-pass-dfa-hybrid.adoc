= 2-pass Sparse Tagged-DFA (TDFA)
:sectnums:

== Concept

DFAs are traditionally poor at extracting submatches because the history of which specific NFA path was taken is lost during state merging. To achieve 100% Go standard library compatibility while maintaining extreme throughput, this project employs a **2-pass Sparse Tagged-DFA (TDFA)** strategy combined with **Strict Priority Burning**.

This strategy separates the concerns of "where is the match" (Pass 1) and "what is inside the match" (Pass 2) to maximize CPU cache efficiency and eliminate all runtime NFA simulation.

== The Core Mandate: NFA is Strictly Prohibited

A core pillar of this engine is that **NFA simulation is never allowed at runtime**. Every decision regarding submatch selection must be "burned into" the DFA transition tables during the compilation phase.

=== Tag-Aware Subsetting (State Multiplexing)
During DFA construction, states are not only defined by their NFA state sets but also by the **history of tags encountered**. If two NFA paths reach the same state with the same priority but different tag updates, the DFA builder splits them into distinct DFA states. 

This ensures that the DFA state itself carries the necessary information to disambiguate paths.

=== Priority Transition Mapping (Burned Logic)
Each DFA transition records a precise mapping: `(input_priority) -> (next_priority, tags)`.
* **Input Priority**: The normalized priority level of the path when entering the state.
* **Next Priority**: The normalized priority level when leaving the state via a specific byte.
* **Tags**: The exact set of capturing group registers to update during this transition.

By pre-calculating these mappings, the engine converts a dynamic NFA-level searching problem into a static table-lookup operation.

== Execution Phases

=== Phase 1: High-Speed Discovery & Recording
* **Objective**: Find the `[start, end]` match boundaries and record the execution history.
* **Mechanism**: Uses a "naked" DFA scan. The `ir.StateID` history is recorded in a lightweight buffer.
* **Performance**: primary search speed matches a pure DFA engine.

=== Phase 2: Deterministic Forward Recap
* **Objective**: Extract capturing group registers along the winning path.
* **Mechanism**:
    1.  Initialize priority `p` from the winning NFA state at `history[start]`.
    2.  For each step `i` from `start` to `end`:
        - Look up the `TransitionUpdate` for `(history[i], input[i])`.
        - Select the update matching the current priority `p`.
        - Apply the burned-in tags and update `p` to the burned-in `next_priority`.
* **Zero Searching**: This pass is strictly $O(n)$ and performs **no dynamic searching**. It only follows the pre-calculated deterministic path.

== Safety & Compatibility

=== Static Compatibility Check
To ensure 100% compatibility, the compiler performs a **Static Compatibility Check**. Patterns that would require non-deterministic runtime logic (like empty-string loops) are rejected at compile time.

== Advantages

* **Strict $O(n)$ Complexity**: Both passes are linear.
* **NFA-Free Architecture**: No runtime NFA traversals, ever.
* **Priority-Burned Precision**: 100% compatibility with Go's standard matching rules.
* **Cache Optimized**: High-speed discovery is never polluted with tagging logic.
