= DFA-First Hybrid (2-Pass Strategy)
:sectnums:

== Concept

DFAs are traditionally poor at extracting submatches (capturing groups). This is because when multiple NFA paths merge into a single DFA state, the history of which path "won" is lost. To extract submatches identical to the Go standard library, this project employs a "2-Pass Strategy."

== The Two Phases

=== Phase 1: Boundary Discovery
* **Objective**: Find the `[start, end]` boundaries of a match as quickly as possible.
* **Mechanism**: Uses a "pure DFA loop" or a loop with minimal tagging.

=== Phase 2: Localized Forward Scan
* **Objective**: Extract precise submatch information within the confirmed boundary.
* **Mechanism**:
  - **Principal DFA Rescan**: Performs a forward scan using the primary DFA to update capture registers (`regs`).
  - **BP-DFA Handover (Greedy Match Resolution)**: When the rescan reaches the confirmed `end` position, and a greedy operator (e.g., `a*`) is involved, the engine hands over the current DFA state to the BP-DFA.
  - **Winning Bit Selection**: The BP-DFA utilizes its bitmask to identify which specific NFA path (bit) reached the `InstMatch` at the exact `end` position. This "Winning Bit" serves as the tie-breaker to resolve the 1-byte ambiguity inherent in DFA-based greedy matches.
  - **Zero-NFA Mandate**: This hybrid approach completely eliminates the need for a backtracking or thread-managed NFA rescan, ensuring consistent $O(n)$ performance.

== Key Technology: Winning Bit Arbitration
The core challenge in DFA submatch extraction is the "Greedy Overrun." In a pattern like `(a*)b`, the DFA state at the match end might contain both a path that matched `a*` and a path that moved to `b`. 

By converting the final DFA state back into a BP-DFA bitmask (`uint64`), the engine can:
1. Identify all NFA paths currently in a "Matching" state.
2. Select the one with the highest priority (lowest bit position) according to Go's leftmost-first semantics.
3. Finalize the capture registers based on this specific winning path.

== Advantages

* **$O(n)$ Guarantees**: All steps stay within a constant multiple of the input length; no backtracking.
* **Memory Bandwidth Savings**: Minimizes writes in Phase 1, keeping CPU caches hot for the data in Phase 2.
* **Full Standard Compatibility**: Guarantees results identical to the `regexp` package, even for complex patterns like `a*(a)`.
