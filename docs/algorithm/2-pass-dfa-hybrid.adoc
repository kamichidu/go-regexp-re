= DFA-Guided 2-Pass Traceback
:sectnums:

== Concept

DFAs are traditionally poor at extracting submatches (capturing groups) because the history of which specific NFA path was taken is lost during state merging. To achieve 100% Go standard library compatibility while maintaining $O(n)$ performance and avoiding NFA thread management (PikeVM), this project employs a **DFA-Guided 2-Pass Traceback** strategy.

This strategy treats the DFA state history as a "safe corridor" through the NFA's potential state space, allowing for a deterministic reconstruction of the unique winning path.

== The Three Phases

=== Phase 1: Consolidated Discovery & Recording
* **Objective**: Find the `[start, end]` match boundaries and record the execution history in a single pass.
* **Mechanism**: Uses a specialized `submatchExecLoop` that performs standard DFA scanning while recording the `ir.StateID` at every byte position into a pre-allocated history buffer.
* **Performance**: By consolidating discovery and recording, the engine minimizes memory bandwidth usage and avoids redundant rescanning of the input.

=== Phase 2: Deterministic Path Reconstruction (Backward Pass)
* **Objective**: Identify the unique sequence of NFA instructions taken for every byte of the match.
* **Mechanism**:
    - **Winning Node Identification**: At the `end` position, the engine inspects the DFA state (which is a set of NFA nodes) and selects the node that reached a `Match` instruction with the highest priority (leftmost-first).
    - **Backward Trace**: Moving from `end` back to `start`, the engine consults the historical DFA states to find the predecessor node.
    - **UTF-8 Awareness**: For multi-byte characters, the engine tracks `ir.NFAPath` (Instruction ID + UTF-8 Trie State). This ensures that the traceback correctly identifies character boundaries and intermediate byte-level transitions.

=== Phase 3: Forward Recap (Tag Application)
* **Objective**: Apply capturing group tags along the confirmed path.
* **Mechanism**:
    - The engine walks forward along the deterministic path reconstructed in Phase 2.
    - **Character Boundaries**: Capture tags are applied only at character boundaries (when the UTF-8 Trie State returns to 0).
    - **Priority Tie-breaking**: If multiple epsilon-transition paths exist between two character-consuming nodes, the engine uses a priority-aware DFS to follow the leftmost path, matching Go's `regexp` behavior exactly.

== Key Technology: NFAPath & MaskStride

=== NFAPath (ID + NodeID)
Standard NFA states are identified by an Instruction ID. To handle UTF-8 without decoding overhead, this engine expands `Rune` instructions into a byte-level trie. An `NFAPath` combines the `InstID` with a `NodeID` (the current position in the trie), allowing the DFA to operate on bytes while maintaining full awareness of rune progress.

=== Multi-stride Bitsets (MaskStride)
To remove the traditional 64-node limitation of bit-parallel engines, the DFA stores NFA node membership in a contiguous array of `uint64` with a "stride" proportional to the pattern size. This allows submatch extraction for patterns with thousands of instructions.

== Advantages

* **$O(n)$ Complexity**: Both recording and traceback are strictly linear with respect to match length.
* **Zero-Allocation**: All buffers (history, path, masks) are managed via a `matchContext` utilizing stack-allocated arrays for common match sizes.
* **Bit-Perfect Compatibility**: Successfully handles complex cases like greedy loops, non-greedy overlaps, and nested empty matches by mirroring Go's priority rules.
