= 2-pass Sparse Tagged-DFA (TDFA)
:sectnums:

== Concept

DFAs are traditionally poor at extracting submatches because the history of which specific NFA path was taken is lost during state merging. To achieve 100% Go standard library compatibility while maintaining extreme throughput, this project employs a **2-pass Sparse Tagged-DFA (TDFA)** strategy.

This strategy separates the concerns of "where is the match" (Pass 1) and "what is inside the match" (Pass 2) to maximize CPU cache efficiency and eliminate dynamic searching during submatch extraction.

== The Three Phases

=== Phase 1: High-Speed Discovery & Recording (Forward Pass)
* **Objective**: Find the `[start, end]` match boundaries and record the execution history.
* **Mechanism**: Uses a "naked" DFA scan that only performs state transitions based on input bytes.
* **Cache Alignment**: The transition table is optimized for 64-byte cache line alignment, with each entry being a 4-byte `ir.StateID`.
* **Zero Overhead**: This pass ignores all capturing group boundaries, ensuring that the primary search speed matches a pure DFA engine.

=== Phase 2: Sparse Tagged Recap (Forward/Backward Walk)
* **Objective**: Apply capturing group tags (registers) along the confirmed winning path.
* **Mechanism**:
    - The engine walks through the recorded `ir.StateID` history within the match boundaries.
    - **Sparse Action Table**: It consults a secondary `tagActionTable`. Unlike the main transition table, this table only contains entries for DFA states that are "tag-sensitive" (i.e., those that contain NFA paths crossing a capture boundary).
    - **Command Execution**: For each edge `(State, Byte)`, a pre-calculated `ActionID` identifies a set of commands (e.g., `regs[0] = current_pos`).
* **Performance**: This pass replaces dynamic NFA closure searching with static table lookups, minimizing branch mispredictions.

=== Phase 3: Final Result Construction
* **Objective**: Format the updated registers into the standard Go `[]int` result.
* **Priority Handling**: Because the `ActionID` is selected during DFA construction based on the highest-priority NFA path, the final submatch indices are guaranteed to match Go's leftmost-first semantics.

== Key Technology: Capture-Aware Sparse Tagging

=== Tag-Sensitive States
During compilation, the engine identifies which DFA states involve transitions that update capturing registers. states that are "tag-neutral" (e.g., matching a long literal prefix or a simple character class) are marked to be skipped during Phase 2, further reducing CPU cycles.

=== Physically Separated Tables
By keeping the `transitions` and `tagActionTable` in separate memory regions, the forward scan (Pass 1) never loads tagging data into the L1 cache. This preserves cache bandwidth for the primary search task.

=== Zero-Allocation recap
All tag update commands are executed using stack-resident register arrays. No heap allocations occur during the recap phase, ensuring stable performance even under high concurrency.

== Advantages

* **$O(n)$ Complexity**: Both discovery and recap are strictly linear.
* **CPU Friendly**: Replaces NFA simulation with simple table-driven register assignments.
* **100% Compatibility**: Mirrors Go's `syntax.Prog` priority rules exactly by baking them into the `ActionID` selection.
* **Memory Efficient**: Sparse tables ensure that tagging overhead is only paid for parts of the regex that actually involve capturing.
