= SWAR Character Class Warp (CCWarp)
:core-tech: SWAR (SIMD Within A Register), Bitwise Parallel, Arithmetic Range Check

== Overview

Standard DFA engines process input one byte at a time, leading to significant overhead in loops, branch mispredictions, and memory latency—especially for patterns like `.*` or `[0-9]+` where the state machine stays in a self-loop.

**SWAR Character Class Warp (CCWarp)** overcomes this bottleneck by treating 64-bit general-purpose registers as SIMD vectors. It processes **8 bytes in parallel** using Pure Go bitwise and arithmetic operations, bypassing the DFA transition table for long repetitions.

== Two-Phase Warping Strategy

The engine applies SWAR optimizations in two distinct phases to maximize throughput across the entire search process.

=== 1. SearchWarp (Pre-filtering)
When the engine is in the initial `searchState` (scanning for a match start), it uses the **SearchWarp** phase to skip "noise" characters. 

- **Hybrid Strategy**: To achieve peak performance, SearchWarp prioritizes Go standard library SIMD-accelerated functions:
    *   **Single Character**: Uses `bytes.IndexByte` (up to 15 GB/s).
    *   **Small Set/Range**: Uses `bytes.IndexAny` with a pre-calculated character string.
    *   **Large/Negated Set**: Falls back to specialized SWAR kernels (e.g., `CCWarpNotSingleRange`) to skip noise 8 bytes at a time.
- **Goal**: Rapidly skip segments of text that cannot possibly start a match.
- **Sharing**: It shares the same kernel definitions as CCWarp but applies **inverted logic**: it skips as long as characters are NOT in the "Start Set".

=== 2. CCWarp (Match Fast-Forwarding)
Once a match has begun and the DFA enters a self-looping state (like the `+` in `[0-9]+`), it switches to the **CCWarp** phase.
- **Goal**: Consume as many matching characters as possible without leaving the current DFA state.
- **Performance**: This phase allows the engine to "swallow" the bulk of the match at memory-bandwidth speeds.

== The Core Principle: Bitwise Parallelism

The foundation of CCWarp is the "Zero-Byte Detection" trick. To check if any byte in an 8-byte word `v` matches a target character `c`, we calculate:

[source,go]
----
diff := v ^ splat(c)
hasMatch := ((diff - 0x0101010101010101) & ^diff & 0x8080808080808080) != 0
----

This allows kernels like `NotEqual` and `AnyExceptNL` to run at the speed of a simple memory copy (~4 GB/s).

== Hierarchical Optimization Kernels

The engine automatically selects the most specialized kernel for both SearchWarp and CCWarp.

[cols="1,2,1,3", options="header"]
|===
| Kernel | Target Pattern | Speed | Strategy
| *Equal* | `a+` | ~5.0 GB/s | Full 64-bit comparison (`v == splat`).
| *AnyChar* | `(?s).*` | ~4.4 GB/s | Total skip. Only checks if bytes are ASCII (Bit 7).
| *AnyExceptNL* | `.*` | ~4.3 GB/s | XOR with `\n` splat.
| *NotEqual* | `[^"]+` | ~3.4 GB/s | XOR with excluded character splat.
| *SingleRange* | `[0-9]+` | ~3.3 GB/s | **Arithmetic Range Check**. Handles continuous ASCII segments.
| *NotSingleRange* | `[^0-9]+` | ~3.2 GB/s | Inverted Arithmetic Range Check.
| *EqualSet* | `[aeiou]+` | ~800 MB/s | **Sub-cube Decomposition**. Chains multiple XOR+Mask operations.
| *NotEqualSet* | `[^ "]+` | ~750 MB/s | Sub-cube chain for negated disjoint sets.
| *Bitmask* | `[\w]+` | ~440 MB/s | 128-bit lookup table applied to 8 bytes batch.
| *NotBitmask* | `[^a-z]+` | ~440 MB/s | Inverted bitmask lookup.
|===

== Advanced Techniques

=== 1. Arithmetic Range Checking
Instead of comparing against a bitmask, `SingleRange` uses arithmetic properties:
A byte `b` is in `[lo, hi]` if `(b + (127 - hi))` does not overflow into Bit 7, and `(b - lo)` does not underflow. By splatting this across 64 bits, we check 8 bytes with just 3-4 instructions.

=== 2. Sub-cube Decomposition
For disjoint sets like `[a-cA-C]`, the engine groups characters that differ by only 1 bit (e.g., 'a' [0x61] and 'A' [0x41] differ by 0x20). By using a bitmask to ignore that specific bit during XOR, we check **two characters with one operation**.

=== 3. Register Pressure & Compiler Optimization
Earlier versions used manual loop unrolling (8x) to maximize throughput. However, the current engine favors **slim `for` loops** over manual unrolling.
- **Why**: Manual unrolling with 8x `uint64` variables consumes too many general-purpose registers, causing the Go compiler to "spill" variables to the stack (memory).
- **Benefit**: By using `for` loops, we reduce active register usage, allowing the compiler to perform its own optimal unrolling based on the target CPU's register availability, resulting in cleaner and faster assembly.

=== 4. Cache Locality Optimization
To prevent performance degradation in complex regex patterns with many states, the core transition data is optimized for the L1 cache.
- **Slim CCWarpInfo**: The structure is kept to **32 bytes**.
- **Heap Fallback**: Large data sets (like 128-bit masks) are offloaded to an `Extra` slice (heap-allocated once).
- **Impact**: This ensures that even DFA tables with hundreds of states can fit within the L2 cache, minimizing memory latency during state transitions.

=== 5. Integration with 3-Pass TDFA
CCWarp is tightly integrated into the 3-Pass execution loop:
- **Strict Self-Loops**: Warping only triggers on states where transitions lead back to themselves without capture tag updates.
- **Naked History**: During the warp, the engine fills the `mc.history` buffer with the state ID for each skipped byte. This allows Pass 2 (Path Selection) to reconstruct the winning path accurately as if it were scanned byte-by-byte.
- **Greedy Boundaries**: When a warp triggers on an accepting state, it records the current index as a potential `bestEnd`. Because it's a self-loop, the `bestStart` and `priority` remain anchored to the initial entry point of that state, preserving Go-compatible leftmost-first-longest semantics.
