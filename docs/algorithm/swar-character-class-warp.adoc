= SWAR Character Class Warp
:sectnums:

== Overview
SWAR (SIMD Within A Register) Character Class Warp is a technique to accelerate the scanning of continuous character classes (e.g., `[a-z]+`) by processing 8 bytes at once using `uint64` registers and bitwise operations, bypassing the standard DFA table lookup loop.

== Design Specification

=== 1. State ID Extension
To support SWAR Warp, the `StateID` (uint32) bit layout is defined as follows:

[source]
----
 32-bit Packed State ID:
 [31] TaggedStateFlag
 [30] AnchorVerifyFlag
 [29] AcceptingStateFlag
 [28] CCWarpFlag (NEW: SWAR Warp possible)
 [27-22] AnchorMask (6-bit EmptyOp)
 [21] WarpStateFlag (Existing: UTF-8 Single-char Warp)
 [19-0] State Index
----

* **CCWarpFlag (Bit 28)**: Indicates that the state is a candidate for 8-byte ASCII character class warping.

=== 2. Compilation: Identifying Warpable States
During the DFA optimization phase, a state $S$ is marked for `CCWarp` if it meets the following criteria:

1. **Self-Loop Existence**: There exists an ASCII byte set $C \subseteq [0, 127]$ such that $\delta(S, c) = S \ (\forall c \in C)$.
2. **Purity Guarantee**:
    * The self-loop transition contains no `RecapEntry` (Tags).
    * The self-loop transition requires no `AnchorVerifyFlag` checks.
3. **Determinism**: $C$ matches one of the predefined SWAR kernels (e.g., a single range `[0-9]` or "any-except-newline" for `.*`).

=== 3. Runtime: Hybrid Warp Strategy
Execution loops (e.g., `fastMatchExecLoop`) attempt skipping in the following order of priority:

[source,go]
----
// Priority 1: SWAR Warp (8-byte parallel)
if state & CCWarpFlag != 0 && i+8 <= len(b) {
    v := binary.LittleEndian.Uint64(b[i:])
    if swarKernel(v, stateParams) {
        i += 8
        for i+8 <= len(b) {
            v = binary.LittleEndian.Uint64(b[i:])
            if !swarKernel(v, stateParams) { break }
            i += 8
        }
        continue
    }
}

// Priority 2: UTF-8 Warp (1-character skip)
if state & WarpStateFlag != 0 {
    i += 1 + ir.GetTrailingByteCount(b[i])
    continue
}

// Priority 3: Standard DFA Transition (1-byte)
state = table[(state << 8) | int(b[i])]
i++
----

=== 4. SWAR Kernel Variants
Optimized kernels are provided for common character class patterns:

* **Kernel 1: Single Range** (`[a-z]`, `[0-9]`)
    * Logic: `((v + 0x7f... - high) | (v - low)) & 0x80... == 0`
* **Kernel 2: ASCII Bitmask** (Complex ASCII sets like `[a-zA-Z0-9_]`)
    * Logic: Verifies all bytes < 128 and checks against a 128-bit bitset.
* **Kernel 3: Any Except Newline** (Optimized for `.*`)
    * Logic: Detects `\n` (0x0A) using zero-byte detection: `(v ^ 0x0A... - 0x01...) & ^(v ^ 0x0A...) & 0x80... == 0`

== Interaction with UTF-8 Warp
* **Complementary**: SWAR Warp handles wide "surfaces" of ASCII data, while UTF-8 Warp handles "points" of multi-byte character boundaries.
* **Safe Fallback**: If SWAR detects an out-of-range byte (e.g., a multi-byte lead byte or a newline), the engine immediately falls back to UTF-8 Warp or standard DFA transitions, ensuring 100% correctness.
* **Performance**: Provides a significant throughput boost for ASCII-heavy payloads, breaking the physical limit of 1-byte-per-transition DFA scanning.

== Simulation Results
Measured throughput for 256MB input on Intel(R) Xeon(R) CPU @ 2.80GHz (`reproduce/sim_swar.go`):

|===
| Method | Throughput | Description

| DFA Baseline
| 0.41 GB/s
| 1-byte table lookup. Dominated by memory latency.

| Simple Byte Loop
| 0.98 GB/s
| 1-byte `if` checks. Better branch prediction but strictly serial.

| **SWAR Range Check**
| **3.61 GB/s**
| **8-byte parallel check via uint64. ~8.8x faster than DFA.**
|===
