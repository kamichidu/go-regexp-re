= Hybrid RLE History for TDFA
:sectnums:
:toc: left

This document describes the **Hybrid Run-Length Encoding (RLE)** strategy for the 3-Pass TDFA engine. It aims to eliminate the $O(n)$ memory bottleneck during submatch extraction without sacrificing the performance of raw byte scanning.

== The Memory Bottleneck
Standard 3-Pass TDFA requires recording the state ID for every byte in Pass 1 to enable backward path reconstruction in Pass 2.
- **Cost**: 4 bytes per input byte (`uint32` state ID).
- **Problem**: A 1GB input requires 4GB of history buffer, leading to frequent `OutOfMemory` errors and cache thrashing.

== Hybrid RLE Strategy
Instead of compressing every transition, the engine applies RLE **exclusively during SWAR (CCWarp) skips**.
- **Observation**: SWAR only triggers on "Pure Self-Loops" where the State ID remains constant across multiple bytes.
- **Hybrid Approach**: 
    * **Raw Mode**: Non-SWAR transitions are recorded as-is (1:1).
    * **Warp Mode**: Consecutive SWAR-skipped bytes are collapsed into a single "Warp Entry" with a length field.

== Data Structure: 32-bit Bit-Packed Entry
To maintain the 32-bit width of `mc.history` entries and support up to 1 million states, we use bit-packing:

[cols="1,1,3", options="header"]
|===
| Bits | Field | Description
| 31 | `WarpMarker` | `1` for RLE/Warp entry, `0` for Raw entry.
| 30-20 | `Length` | Number of bytes covered (1 to 2047).
| 19-0 | `StateID` | The DFA State ID (supports up to 1,048,575 states).
|===

=== Bit Layout Diagram
[bytefield]
----

+--+-------------+---------------------------------------------+
|W |   Length    |                   State ID                  |
+--+-------------+---------------------------------------------+
----

== Algorithm Refinement

=== Pass 1: Recording with Accumulation
When the SWAR kernel skips $N$ bytes (e.g., 8 bytes):
1. Check the last entry in `mc.history`.
2. If `WarpMarker == 1`, `StateID` matches, and `Length + N < 2048`:
   - Increment `Length` by $N$.
3. Otherwise:
   - Append a new Warp Entry: `1 | (N << 20) | sidx`.

=== Pass 2: Backward Path Selection (The Warp Jump)
Pass 2 traverses the history in reverse. With RLE, it can "jump" over large repetitions:
[source,go]
----
for i := len(mc.history) - 1; i >= 0; i-- {
    entry := mc.history[i]
    if (entry >> 31) != 0 {
        length := (entry >> 20) & 0x7FF
        sidx := entry & 0xFFFFF
        // Jump backward by 'length' bytes
        currentPos -= int(length)
        // Path logic remains constant across the warp (self-loop)
    } else {
        currentPos--
    }
}
----

=== Pass 3: Forward Recap (Licking)
Since SWAR only triggers on states **without tag updates**, Pass 3 can safely skip the entire warp duration:
- Update `regs` only at the start and end boundaries of the warp.
- Increment `currentPos` by `length` in a single step.

== Expected Benefits

| Benefit | Impact |
| :--- | :--- |
| **Memory Reduction** | Up to **99.9%** reduction for repetitive patterns (e.g., `.*`, `a+`). |
| **Pass 2/3 Speed** | **8x to 2000x faster** for warped regions due to reduced loop iterations. |
| **Cache Locality** | History data fits in L1/L2 cache even for large inputs, reducing DRAM traffic. |
| **Zero Regression** | For random/noise data where SWAR doesn't trigger, memory and speed remain identical to current raw implementation. |

== Constraints & Safety
- **Length Cap**: The `Length` field is capped at 2047. If a repetition exceeds this, a new Warp Entry is started.
- **State ID Limit**: The engine is limited to 1,048,575 states. Patterns exceeding this will fallback to structural rejection or "Too Large" errors.
- **Self-Loop Only**: RLE MUST NOT be applied if the self-loop transition contains any tag updates or anchor verifications.
