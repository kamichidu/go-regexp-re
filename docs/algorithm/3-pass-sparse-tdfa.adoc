= 3-Pass Sparse TDFA (Next-Gen Submatch Extraction)
:sectnums:

== Concept

DFAs are traditionally poor at extracting submatches because the history of which specific NFA path was taken is lost during state merging. To achieve 100% Go-compatible submatch extraction with $O(n)$ time complexity, this engine employs a **3-Pass Sparse TDFA** architecture.

The core idea is to decouple match discovery from boundary extraction, using a **Naked DFA** for speed and a **Recap Table** for precision.

== The 3-Pass Workflow

=== Phase 1: Naked Discovery (Forward Scan)
*   **Objective**: Determine the match boundaries `[start, end]` and record the state history.
*   **Mechanism**: Uses a **Naked DFA** for match discovery.
    *   **State Identity via MatchPriority**: To prevent incorrect match boundary extensions (e.g., `aa|a` matching `aaaa` as `[0 4]`), Phase 1 DFA states include the **MatchPriority** (the highest priority NFA path in the state) in their identity key. This ensures that states with identical NFA IDs but different leftmost-first implications are physically separated.
*   **Output**: A successful match window `[start, end]`, the final `MatchPriority`, and a `history []uint32` of DFA state IDs.

=== Phase 2: Path Identity Selection (Backward Trace)
*   **Objective**: Determine the unique "winning path" from `start` to `end` based on Go's leftmost-first rules.
*   **Mechanism**: Performs a **backward trace** from the `end` position.
    1.  Initialize `PathID` from the `MatchPriority` of the final accepting state.
    2.  Work backward to the `start` position by matching `entry.NextPriority` in the `RecapTable`.
*   **Priority Delta Restoration**: During backward traversal, the engine restores the previous path identity using **Priority Deltas**. It adds the `BasePriority` (shift amount) stored in the `TagUpdate` to the `entry.InputPriority`.
*   **Output**: A sequence of `PathIDs` (recorded in `mc.pathHistory`) that uniquely identifies the winning NFA path.

=== Phase 3: Pure Forward Recap (Licking)
*   **Objective**: Extract capturing group registers along the confirmed winning path.
*   **Mechanism**:
    1.  Starts at the `start` position and applies initial tags from `StartUpdates`.
    2.  Iterates forward from `start` to `end`. At each step $i$, it select the unique transition where `entry.InputPriority == pathID - BasePriority` and `entry.NextPriority == nextPathID`.
    3.  Applies **delta tags** from the transition.
*   **Go-Compatible Tagging Rules**:
    *   **Start Tags** (even bits): Recorded only on the first encounter (leftmost).
    *   **End Tags** (odd bits): Updated on every encounter (licking).

== Technical Implementation Details

=== Transition-Resident Delta Tags
To prevent submatch boundary corruption, tags are never stored within DFA states.

*   **Delta-Only**: Transitions only store the tags that were newly encountered during that specific byte consumption.
*   **Priority Delta Propagation**: During the forward scan (Phase 1), the engine tracks priority using `BasePriority` deltas. This prevents absolute priority accumulation from corrupting search restart logic.

=== Match Integrity Guard (Static Analysis)
Patterns that match empty strings in a loop (Epsilon Cycles) or contain ambiguous explicit empty branches in captures are rejected at compile time. This ensures that every execution path in the TDFA has a deterministic path identity sequence.

== Advantages

*   **NFA-Free Execution**: No runtime NFA simulation or backtracking.
*   **Strict $O(n)$ Complexity**: All three passes are linear relative to input length.
*   **Zero-Allocation Submatch Extraction**: Operates entirely on pre-allocated buffers and stack-resident state.
*   **100% Go Compatibility**: Guarantees leftmost-first match results identical to `regexp`.
