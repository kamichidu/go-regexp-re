== Overview

`go-regexp-re` is a high-density, high-performance Deterministic Finite Automaton (DFA) engine implemented in **Pure Go**.
The project's primary objective is to push the boundaries of regular expression processing speed within the Go ecosystem while maintaining 100% compatibility with the standard `regexp` library's `leftmost-first` matching semantics.

By combining $O(n)$ time complexity with advanced SIMD acceleration and bit-parallel processing, this engine explores the theoretical and physical limits of string matching in a memory-safe, native Go environment.

== Core Architectural Innovations

=== 1. xref:algorithm/0-pass-literal-bypass.adoc[0-Pass (Literal Bypass) & Capture Template]
An $O(1)$ bypass strategy for pure literals and anchored fixed strings, extracting submatches without driving the state machine.

=== 2. xref:algorithm/rejected-bit-parallel-dfa.adoc[Bit-parallel DFA (BP-DFA) - Rejected]
A previously explored execution path for small patterns. While conceptually fast, it was rejected in favor of the more flexible and scalable **SWAR Character Class Warp** strategy.

=== 3. xref:algorithm/guarded-simd-warp.adoc[Anchor-Aware Guarded SIMD Warp]
A speculative SIMD skipping logic designed to maintain high-speed scanning even in the presence of anchors.

=== 4. xref:algorithm/multi-pass-sparse-tdfa.adoc[Multi-Pass Sparse TDFA (Execution Pipeline)]
A deterministic multi-pass execution strategy that decouples heuristic skipping, deterministic discovery, and precise submatch extraction to eliminate state explosion and history pollution.

=== 5. xref:algorithm/memory-efficient-construction.adoc[Memory-Efficient DFA Construction]
An allocation-free and pointer-free construction algorithm designed to scale to massive pattern sets within strict memory constraints.

=== 6. xref:algorithm/implementation-techniques.adoc[High-Efficiency Implementation Techniques]
Advanced Go-specific optimizations, including manual loop monomorphization and byte-level UTF-8 trie expansion.

=== 7. xref:algorithm/ast-optimization.adoc[AST-Level Optimization & Factoring]
Pre-compilation AST rewriting and common sub-expression factoring to minimize state divergence and memory footprint.

=== 8. xref:algorithm/multibyte-warp-optimization.adoc[Multi-byte Warp Optimization & State Explosion Protection]
A dual-purpose strategy for deterministically skipping multi-byte UTF-8 sequences and protecting against NFA path set divergence in loop structures.

=== 9. xref:algorithm/swar-character-class-warp.adoc[SWAR Character Class Warp (Hybrid Strategy)]
The engine's primary performance layer. It combines **SearchWarp** (SIMD-accelerated noise skipping) and **CCWarp** (SWAR-based match continuation) to achieve multi-gigabyte throughput for common patterns and repetitions (e.g., `a+`, `[0-9]+`).

=== 10. xref:algorithm/rejected-prefilter-indexany.adoc[Character Class Pre-filtering via bytes.IndexAny - Rejected]
A proposal to use `bytes.IndexAny` as a noise-skipping pre-filter. It was rejected because the engine's optimized DFA execution loops proved to be faster than the standard library's `IndexAny` for most common character sets.

=== 11. xref:algorithm/hybrid-rle-history.adoc[Hybrid RLE History (Submatch Memory Optimization)]
An advanced 32-bit bit-packing strategy that applies Run-Length Encoding (RLE) during SWAR skips. It reduces history buffer consumption by up to 99.9% for repetitive patterns and enables ultra-fast "Warp Jumps" during backward path selection in the Multi-Pass pipeline.

== Design Philosophy

* **Pure Go (Zero CGO)**: Achieving maximum performance without sacrificing portability or introducing the overhead of C-interop.
* **Optimization for Physical Constraints**: Strategic alignment with CPU cache lines, pipeline efficiency, and Go's runtime characteristics.
* **Deterministic Guarantees**: Ensuring $O(n)$ time complexity to prevent ReDoS (Regular Expression Denial of Service) and provide predictable latency.
* **Precision Mandatory**: Performance enhancements never compromise the exact matching behavior defined by Go's standard `regexp` package.

== Explicit Design Trade-offs & Incompatibilities

The project makes deliberate sacrifices to achieve its performance targets:

=== 1. Shift to Compilation
Complexity is pushed from execution to the `Compile` phase. The engine performs exhaustive NFA-to-DFA conversion, DFA minimization, and AST optimization during compilation. While this ensures ultra-fast execution, it results in longer compilation times compared to NFA engines.

=== 2. Memory-Centric Architecture
Performance is essentially "bought" with memory. A single DFA state occupies 1KB (256 * 4 bytes). A pattern with thousands of states can consume significant memory (e.g., 64MiB+). While the engine provides `Memory-Efficient Construction` to manage this, memory consumption will always be higher than the standard library.

=== 3. State Explosion Protection (MaxMemory)
DFA construction can lead to an exponential explosion of states for patterns with high ambiguity (e.g., multiple overlapping wildcards). To prevent system exhaustion (OOM), `go-regexp-re` enforces a configurable memory limit.
- **Default Limit**: 64MiB.
- **Failure Mode**: If a pattern exceeds the limit, the compiler returns an error: `regexp: pattern too large or ambiguous`.
- **Mitigation**: Users must either simplify the pattern or explicitly increase the `MaxMemory` via `CompileOptions`.

=== 4. No POSIX Semantics
POSIX regex semantics are explicitly unsupported. Following POSIX would break the $O(n)$ guarantee and introduce complex branching that would cripple the DFA's throughput.

=== 5. Byte-Oriented vs. Rune-Oriented
By scanning raw bytes, we achieve maximum CPU throughput. However, this means we cannot emulate the standard library's behavior of matching invalid UTF-8 bytes with `\uFFFD`. We treat all inputs as a raw byte stream.
