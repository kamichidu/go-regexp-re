go-regexp-re is a pure Go byte-oriented DFA regex engine. It is designed for extreme throughput and drop-in compatibility with the standard `regexp` package while maintaining $O(n)$ time complexity.

== Technical Characteristics

* **Pure Go (No CGO, No Assembly)**: Implementation is written entirely in native Go. It achieves extreme performance through manual loop monomorphization and compiler-friendly optimizations, avoiding architecture-specific assembly code and CGO overhead for maximum portability and maintainability.
* **Byte-Oriented Scanning**: Input is processed as a raw byte sequence (`[]byte`) to avoid the overhead of UTF-8 rune decoding and maximize CPU pipeline efficiency.
* **Deterministic Finite Automaton (DFA)**: Patterns are converted into deterministic transition tables to ensure constant-time processing per input byte.
* **Drop-in Compatibility**: The exported API follows the patterns established by the Go standard `regexp` package.

== Performance Model

The engine achieves high throughput by offloading computation to the compilation phase and utilizing SIMD/bit-parallel execution paths. This performance model is optimized for long-lived regex objects where the construction cost is amortized over many execution cycles.

== Architectural Implementation

The engine employs a modular architecture designed to separate the compilation phase from the high-performance execution phase.

1. **Compilation Layer**: Managed within `internal/ir/`. This layer handles state space exploration, NFA to DFA conversion (Subset Construction), and transition table generation.
2. **Execution Layer (Runtime)**: Located in the root package. It utilizes manually monomorphized hot-loops to achieve zero-overhead matching, selecting the most efficient kernel based on pattern characteristics.
3. **AST Optimization**: Managed within `syntax/`. Performs project-specific simplifications and optimizations before the pattern is compiled into an intermediate representation.

=== Core Components

- **Strategy Dispatcher**: Automatically routes patterns to the optimal execution path (Literal Bypass, Bit-parallel, or Pure DFA).
- **3-Pass Sparse TDFA**: A deterministic submatch extraction engine that guarantees $O(n)$ time complexity by utilizing pre-calculated transition-resident tags.
- **SIMD/SWAR Warping**: High-speed segment skipping that fast-forwards through non-matching data using CPU-native vector instructions.

== Usage

The default memory limit for DFA construction is **64MiB**. For patterns requiring larger state spaces, use `CompileWithOptions`.

[source,go]
----
import (
    regexp "github.com/kamichidu/go-regexp-re"
)

func main() {
    // Basic usage
    re := regexp.MustCompile(`[a-zA-Z_][a-zA-Z0-9_]*`)

    // Custom options (e.g., higher memory limit for complex patterns)
    reComplex, err := regexp.CompileWithOptions(`pattern`, regexp.CompileOptions{
        MaxMemory: 256 * 1024 * 1024,
    })
}
----

=== Best Practice: Long-lived Objects

Regex compilation in this engine is computationally expensive. To amortize the construction cost, it is recommended to compile patterns once and reuse the resulting objects.

[source,go]
----
var identRe = regexp.MustCompile(`[a-zA-Z_][a-zA-Z0-9_]*`)

func IsIdentifier(s string) bool {
    return identRe.MatchString(s)
}
----

== Compatibility & Constraints

`go-regexp-re` inherits the core philosophy of the https://pkg.go.dev/regexp[Go standard regexp package^]: prioritizing $O(n)$ performance and safety over features that require backtracking.

For a comprehensive breakdown of our validation strategy and architectural boundaries, refer to the **xref:docs/compatibility-policy.adoc[Compatibility Policy]**.

=== Shared with Go Standard Library

Like the standard library (RE2-based), this engine enforces constraints necessary to guarantee linear-time execution:

1. **No Backreferences**: Features like `(a)\1` are strictly unsupported as they would require exponential-time backtracking.
2. **ASCII-Only Character Classes**: Common Perl character classes and boundaries are restricted to the ASCII range, matching standard Go behavior:
   - `\d` is `[0-9]`, `\s` is `[\t\n\f\r ]`, `\w` is `[a-zA-Z0-9_]`.
   - `\b` and `\B` are calculated based on the ASCII `\w` definition.
3. **Leftmost-First Rule**: We strictly follow Go's standard matching priority. POSIX "longest-match" semantics are not supported.

=== Engine-Specific Architectural Constraints

1. **Strict Capture Determinism**: Patterns that introduce non-deterministic tag updates (e.g., `(|a)*`, `(a|)`) are rejected at compile-time with `regexp.UnsupportedError`.
2. **Submatch Precision**: While overall match boundaries (indices 0 and 1) are 100% compliant, internal submatch boundaries for highly ambiguous patterns (e.g., `a*(a)`) may occasionally deviate.
3. **Byte-Level Semantics**: The engine matches physical bytes and does not support standard `\uFFFD` replacement semantics for invalid UTF-8 sequences.

== Roadmap

1. **Repetition Priority Resolution**: Continued alignment of greedy loop submatch boundaries with standard library behavior.
2. **Construction Phase Optimization**: Reduction of compilation time and memory footprint for complex patterns.
3. **Concurrent Scan Support**: Formal verification and benchmarking of concurrent execution across multiple goroutines.
4. **Postal Code Stress Test**: Validating scaling and performance with extremely large pattern sets (100,000+ literals).
5. **Fixed-length Assertions**: Implementation of efficient lookahead and lookbehind assertions within the DFA framework.
6. **Memory Efficiency**: Optimizing the DFA transition table storage for massive alternations to reduce memory pressure.


== Acknowledgments

`go-regexp-re` owes its core concept and inspiration to the legendary Perl module **`Regexp::Assemble`**.

Years ago, the profound elegance and power of `Regexp::Assemble` left a lasting impression on me. This project is an attempt to evolve that same spirit within the modern Go ecosystem, pushing the boundaries of DFA-based matching to its physical limits.
