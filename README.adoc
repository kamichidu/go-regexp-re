====
This project is in **Beta**. Architectural changes and bug fixes are ongoing. Use in production environments should be evaluated accordingly.
====

go-regexp-re is a pure Go byte-oriented DFA regex engine. It is designed for extreme throughput and drop-in compatibility with the standard `regexp` package while maintaining $O(n)$ time complexity.

== Technical Characteristics

* **Pure Go**: Implementation is written entirely in Go without CGO dependencies.
* **Byte-Oriented Scanning**: Input is processed as a raw byte sequence (`[]byte`) to avoid the overhead of UTF-8 rune decoding and maximize CPU pipeline efficiency.
* **Deterministic Finite Automaton (DFA)**: Patterns are converted into deterministic transition tables to ensure constant-time processing per input byte.
* **Drop-in Compatibility**: The exported API follows the patterns established by the Go standard `regexp` package.

== Performance Model

The engine achieves high throughput by offloading computation to the compilation phase and utilizing SIMD/bit-parallel execution paths. This performance model is optimized for long-lived regex objects where the construction cost is amortized over many execution cycles.

== Architectural Implementation

1. **Compilation Cost**: The `Compile` phase involves state space exploration and transition table construction. For complex patterns, compilation time and memory usage are higher than the standard NFA-based implementation.
2. **Memory Requirements**: The resulting DFA transition tables occupy memory proportional to the number of states and the alphabet size.
3. **Memory Limits**: A configurable `MaxMemory` parameter is enforced during compilation to prevent unbounded memory allocation.
4. **Byte-Level Semantics**: The engine matches physical bytes. It does not support standard `\uFFFD` semantics for invalid UTF-8 sequences.

== Usage

The default memory limit for DFA construction is **64MiB**. For patterns requiring larger state spaces, use `CompileWithOptions`.

[source,go]
----
import (
    regexp "github.com/kamichidu/go-regexp-re"
)

func main() {
    // Basic usage
    re := regexp.MustCompile(`[a-zA-Z_][a-zA-Z0-9_]*`)

    // Custom options
    reComplex, err := regexp.CompileWithOptions(`pattern`, regexp.CompileOptions{
        MaxMemory: 256 * 1024 * 1024,
    })
}
----

=== Best Practice: Long-lived Objects

Regex compilation in this engine is computationally expensive. To amortize the construction cost, it is recommended to compile patterns once and reuse the resulting objects across multiple execution cycles.

* **Recommendation**: Utilize package-level variables or singletons to store compiled regex objects. Avoid invoking `Compile` or `MustCompile` within hot loops or performance-critical paths.

[source,go]
----
var identRe = regexp.MustCompile(`[a-zA-Z_][a-zA-Z0-9_]*`)

func IsIdentifier(s string) bool {
    return identRe.MatchString(s)
}
----

== Compatibility & Constraints

The following constraints apply to maintain $O(n)$ time complexity and DFA determinism:

1. **No Backreferences**: Backreferences (e.g., `(a)\1`) are not supported.
2. **No POSIX Semantics**: Standard Go (leftmost-first) semantics are used; POSIX matching is unsupported.
3. **API Subset**: Functions that require features incompatible with DFA execution are omitted.

=== Known Issues: Submatch Precision

Submatch extraction is implemented via a 3-pass strategy. The following cases may exhibit boundary offsets of $\pm 1$ byte compared to the standard library:

* **Greedy Loop Boundaries**: Patterns such as `a*(a)` or `(.*)a` when matching repeated characters.
* **Complex Nested Groups**: Certain configurations of nested optional groups like `a*(|(b))c*`.

Refinements to the priority resolution logic are ongoing to address these edge cases.

== Implementation Details

Refer to the internal documentation for details on the algorithms used:

* **xref:docs/algorithm.adoc[Technical Documentation: Algorithms]**

Summary of components:
- **0-Pass (Literal Bypass)**: SIMD-based matching for fixed-string patterns.
- **Bit-parallel DFA**: Bitwise state transitions for small instruction sets.
- **Guarded SIMD Warp**: Accelerated skipping for patterns with fixed prefixes.
- **3-Pass Sparse TDFA**: Deterministic submatch extraction consisting of:
    1. **Naked Discovery**: Identifies match boundaries.
    2. **Backward Trace**: Reconstructs the NFA path from final priority.
    3. **Forward Licking**: Applies register updates using transition-resident delta tags.

== Roadmap

1. **Repetition Priority Resolution**: Alignment of greedy loop submatch boundaries.
2. **Construction Phase Optimization**: Reduction of compilation time and memory footprint.
3. **Concurrent Scan Support**: Verification of concurrent execution across goroutines.
4. **Postal Code Stress Test**: Scaling to large pattern sets (100,000+ patterns).
5. **Fixed-length Assertions**: Support for lookahead/lookbehind.

== Acknowledgments

`go-regexp-re` owes its core concept and inspiration to the legendary Perl module **`Regexp::Assemble`**.

Years ago, the profound elegance and power of `Regexp::Assemble` left a lasting impression on me. This project is an attempt to evolve that same spirit within the modern Go ecosystem, pushing the boundaries of DFA-based matching to its physical limits.
